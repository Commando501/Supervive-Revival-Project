=== SESSION 32 — RPC SIGNATURE FULLY SOLVED via self-replay harness ===

Session 32 delivered the complete ServerVerifyViewTarget signature. Key
innovation: a **stub-side self-replay harness** that walks the injected
UFunction's properties over a BitReader wrapping the captured 2298-bit RPC
arg struct (from session 25's bunch capture) — zero client involvement,
instant per-trial feedback.

--- Self-replay harness (unreal-stub/Source/Loki/Loki.cpp) ---

At OnPostEngineInit, after UFunction injection, the stub:

1. Embeds the 293-byte captured bunch bytes inline (kBunchBytes).
2. Extracts bits 41..2338 (the 2298-bit RPC arg struct) into a fresh
   LSB-packed byte buffer.
3. Wraps in FBitReader.
4. Walks UFunction ChildProperties. For each param:
   - FObjectProperty: skip 18 bits manually (measured session 30
     via real-client SetOverflowed diagnostic).
   - All other properties: call NetSerializeItem with Map=nullptr
     (safe for bool, int, uint32, byte, FString — these don't touch
     PackageMap).
5. Logs Pos before/after each property + FString content for StrProperty.

--- Semantic interpretation ---

SUPERVIVE overrode ServerVerifyViewTarget to include a per-sub-level state
report from client → server. Each of the 5 elements is a sub-level of
LVL_LobbyV2_Persistent (PartyMenu, HeroSelect_Skylands, BattlePass,
Lighting, Armory). The 45-bit state struct per sub-level (uint32 + uint8
+ 5 bools) most likely encodes:

  * uint32   — LoadRequestID / StreamingID / handle
  * uint8    — LoadState enum (Unloaded, Loading, Loaded, Visible, etc.)
  * 5 bools  — flag bits (bIsVisible, bIsPendingLoad, bWasRequested, ...)

All observed U32/U8/bool values in the captured payload were ZERO —
suggests either default-initialized state OR the client had not populated
these fields yet at the moment of capture (early in LobbyV2 travel).

The last element has 4 fewer bool bits than the others (bool×1 vs bool×5).
This may be a bit-packing efficiency (last element in a fixed-count list
omits trailing bools), or my inferred struct is slightly off — the last
struct might just have 1 bool where the others have 5. Either way, the
byte alignment works.

--- What the harness does NOT test ---

The self-replay is a PROPERTY-WALK simulation, not a live UE RPC dispatch.
A real client bunch will additionally exercise:

  1. FRepLayout::ReceivePropertiesForRPC (rather than direct
     Property::NetSerializeItem calls).
  2. UPackageMap.SerializeObject for the actual AActor* NetGUID resolution.
  3. Reader.PackageMap->ResetTrackedGuids and related channel state.

But the raw bit-stream layout MUST match what our signature declares — and
that we've fully verified.
