--- What's needed for session 39+ ---

The new blocker: client goes silent for 51 seconds, then server-timeout closes.
Root cause: the client's post-Join flow expects SOMETHING from the server
(most likely PC / GameState replication) before it will send its
ServerVerifyViewTarget RPC or drive its own state machine forward. Our full
suppression makes the server invisible.

Two paths forward:

  A. Server-side keep-alive to prevent timeout while we investigate.
     Simplest — just send heartbeat bunches on the control channel so the
     server-side idle-timeout tracker sees activity. Buys diagnostic time,
     doesn't solve the actual problem.

  B. Option B (proper fix): runtime FProperty injection to make our
     APlayerController's replicated schema MATCH SUPERVIVE's client
     expectation. Same primitive as sessions 27–32's UFunction param
     injection, applied to top-level UProperties on APlayerController /
     AGameStateBase / APlayerState. Requires reverse-engineering SUPERVIVE's
     schema via usmapdump.exe schema (session 22 established that route
     works). Then un-suppress those classes.

  C. Half-Option-B (compromise): keep suppression, but ADD one hand-crafted
     bunch that satisfies whatever minimum state check the client makes
     post-Join. Requires reading the client's ReceivedJoin → next state
     transition code to identify the trigger.

Recommendation for session 39: A first (10 minutes of engineering — confirms
timeout is really the only remaining issue, buys observation time). Then
survey the client's post-Join code path (via usmapdump.exe / disassembly) to
figure out what state it's waiting for. That informs whether B or C is the
right investment.
