    +0x2976FE6   mov rcx, qword ptr [rdi]             ; rcx = array.GetData()

    +0x2976FF0   mov qword ptr [rcx+8*rsi], rax       ; array[rsi] = rax  <-- CRASH

Classic TArray::Add pattern:

    UObject* this;
    int32 idx = arr.Num();
    arr.SetNum(idx + 1);
    if (idx + 1 > arr.Max()) arr.Grow();
    arr[idx] = someValue;  // <-- crashes because arr.GetData() points at .rdata

Failed write target 0x00007ff68a55b368 = base + 0x7ADB368 — an address
inside the exe's read-only data or code segment. This is what you get when
a TArray's data pointer holds a stale/default value pointing into .rdata
instead of a heap allocation.

--- Interpretation ---

The TArray at `[this+0x50]` on some UObject has:
- .Num incremented to > 0 (either by SetNum here or previously)
- .Max large enough that Grow() didn't fire
- .Data pointing at .rdata (never actually allocated a heap buffer)

This inconsistency between .Num/.Max and .Data usually happens when a
UObject was constructed but its TArray wasn't properly initialized before
Add(). Session 38's full suppression left many UObjects in an "exists but
missing expected server-side data" state. One of them has a TArray field
that later code assumes was initialized.

Session 39 does NOT identify which specific UObject/field. That requires
Ghidra/IDA symbols or a targeted Fname-based search of the exe for the
containing class name.

--- What session 40+ needs ---

Two orthogonal paths:

- **A** (surgical): identify the specific UObject / class whose TArray
  is being Add()'d to at crash time, then feed it enough state via our
  stub so the array is properly initialized before menu code touches it.
  Requires either Ghidra symbols or targeted usmapdump `wstrings` / `disasm`
  work at the crash callers (+0x2984F45, +0x2983C27, +0x29838BE, +0x29CCD3E)
  to identify the module.

- **B** (breadth): reverse the FClassNetCache divergent classes by injecting
  matching UProperties on the stub-side classes and un-suppressing them.
  That way the client receives the state it expects and the "missing
  initialization" problem never arises. Same primitive as sessions 27-32's
  UFunction param injection, applied to top-level UProperties on the
  session-20 divergent classes. Substantial engineering — needs
  usmapdump.exe schema output for each class.

- **B-lite**: pick JUST APlayerController (the most-relied-on class) for
  the schema-injection treatment. Get a real PC replicated to the client
  and see whether the crash mode changes.

Recommendation: A first (cheap information — surface the crash class name),
then B-lite (if A reveals the culprit is PC-owned state). Full B is the
long-term fix but expensive.
