--- Class identification: CONFIRMED ---

The class at vtable RVA +0x7B9E188 is:
- **Only reachable when D3D12Core.dll is loaded** (renderer = D3D12)
- **Called heavily by D3D12Core.dll** (minidump call stack shows
  ~10 D3D12Core.dll frames interleaved with SUPERVIVE frames)
- Object size 15,504 bytes; multi-inheritance
- Parent class (232 bytes, vtable +0x7B9DE88) has method
  emitting assertion logs referencing `D3D12CommandList.cpp`
- Log strings adjacent to the vtable reference `GraphicsCommandList`,
  `CopyCommandList`, `D3D12CommandContext.cpp`, `D3D12Residency.h`

All signals now agree: the class is a **UE 5.4 D3D12 RHI class**,
specifically one that D3D12Core.dll calls into (device child /
command list wrapper / pipeline state object / similar).

Most likely candidate: **FD3D12CommandContext** or **FD3D12CommandList**
descendant. The parent-class hypothesis stands: parent = FD3D12CommandList
(232 bytes), our class inherits from it (15,504 bytes).

--- The crash scenario, restated with confidence ---

D3D12 mode:
1. Client boots, initializes D3D12 RHI, loads D3D12Core.dll
2. Client connects to our stub, reaches ReceivedJoin
3. Client fetches menu data (25 heroes, wallet) via ags Messenger
4. Client tries to record initial render commands for the menu UI
5. Somewhere in the render-command setup, `slot 1` (RVA 0x2976F70)
   of our D3D12 class fires
6. The `subarray at index 2` inside the "slot parent" object (vtable
   +0x7B9DC48, 0x160 bytes) has count=1, cap=4, but Data still points
   at .rdata const-default (never allocated)
7. The write `array[0] = value` targets read-only memory at
   RVA +0x7ADB368
8. EXCEPTION_ACCESS_VIOLATION writing address 0x7FF68A55B368

The uninit TArray belongs to the 0x160-byte slot-parent instance (a
sibling class in the D3D12 RHI). Our target class OWNS that instance
inline. Some earlier initialization step normally populates the TArray's
Data pointer via an initial `Add()` (which triggers Grow()), but our
server-side suppression causes that init to be skipped.
