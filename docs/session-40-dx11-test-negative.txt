--- Interpretation ---

The FD3D12CommandContext identification in `session-40-path-a1-D3D12-
identified.txt` was TOO SPECIFIC. Some possibilities:

1. **The class isn't renderer-specific**. The log strings referencing
   D3D12CommandList.cpp happen to be near the vtable in `.rdata`, but
   the actual vtable belongs to a different (more general) class.
   The MSVC linker places `.rdata` strings roughly in source-file order,
   so nearby strings often but not always belong to the class whose
   vtable is nearby.
2. **The D3D12 subsystem is still initialized in DX11 mode** — some
   UE 5.4 subsystems (RayTracing, GPU capture, DirectStorage, etc.)
   may create D3D12 contexts even when the primary renderer is DX11.
   If those subsystems' contexts hit our crash path, DX11 mode doesn't
   help.
3. **The crash is post-RHI, in a subsystem that both DX11 and D3D12
   modes use**. The crash-fn is at RVA 0x2976F70; the log-string
   parent-slot-2 was at RVA 0x296F2C0. Only slot 2 (out of 12 primary
   slots) proved to have D3D12 log strings — the class might have
   D3D12-conditional code paths in specific slots but be a more
   general class overall.

--- What this tells us about the actual root cause ---

Time-to-crash is renderer-mode-dependent (24s in DX11, 64s in D3D12).
That's consistent with "the crash class is in some subsystem that ticks
+ tries to Add() to a subscriber list, and the tick rate / activation
differs between renderers". It's NOT consistent with "the crash is a
one-time hit in the D3D12 command context init".

--- Committed Path A1 finding needs REVISION ---

`session-40-path-a1-D3D12-identified.txt` overstates the confidence.
The evidence supports:
- The crash class's vtable is in `.rdata` NEAR D3D12RHI code strings
- The crash class inherits from a class whose slot 2 assertion logs

The evidence does NOT support:

More accurate identification: **the class is likely still an RHI-related
or graphics-subsystem class, but not renderer-specific enough to be
avoided by DX11 mode. It might be a base class that has both DX11 and
D3D12 concrete subclasses, or a subsystem that both use.**

--- What to try next (session 41+) ---

Given DX11 fails equally, the useful next tests are:

- **`-vulkan` cmdline** — try a completely different RHI. If Vulkan
  also crashes, the class is definitively renderer-independent.
- **`-nullrhi`** — no rendering at all. If the client boots and reaches
  ReceivedJoin AT ALL without a renderer, and reaches menu-load state
  without crashing, then the crash IS renderer-related just not
  D3D12-specific.
- **Investigate WHAT SPECIFICALLY differs between D3D12 (64s survival)
  and DX11 (24s survival)** — the 40-second delta tells us the crash-
  causing code has different frequency of invocation under the two
  renderers. Could be per-frame vs per-tick differences, could be
  render-command-list generation cadence, etc. Doesn't help identify
  the class directly but gives us a knob.
- **Test the crash under DIFFERENT graphics quality / resolution
  settings** — some UE renderer paths only activate at specific
  quality levels.

**PATH B-LITE re-evaluation:** Given the crash isn't renderer-tied, it
IS still worth trying Path B-lite (schema-inject replicated fields on
PC + un-suppress) as the next investment. The 40-second timing delta
between D3D12 and DX11 modes suggests the crash-cause TArray is populated
by some subscriber that ticks at different rates per RHI. If the
subscribers are UE Actor components (which they often are for rendering-
related work), then AActor replication state might indirectly fix the
init sequence.
