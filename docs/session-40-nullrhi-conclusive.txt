=== SESSION 40 NULLRHI TEST — CRASH IS DEFINITIVELY NOT RENDERER-RELATED ===

Final renderer-variant test. Prediction was that -nullrhi (no rendering
hardware interface at all) might not even boot, or might skip the crash
if it's renderer-tied.

**Actual result: client BOOTED, reached ReceivedJoin, LOADED MENU (25
heroes fetched, wallet fetched), and CRASHED at ~32 seconds post-Join
via Sentry HandleBeforeCrash.**

with `-nullrhi` swapped in. Same session-38 stub + session-38 code state.

**Since -nullrhi has no rendering at all yet the crash still happens,
the crash class is DEFINITIVELY NOT renderer-related.**

The FD3D12CommandContext identification in
`session-40-path-a1-D3D12-identified.txt` is fully retracted. The
log-string evidence (D3D12CommandList.cpp / GraphicsCommandList /
CopyCommandList) was misleading — those strings are near our target
vtable in `.rdata` but the vtable does not belong to a D3D12 class.
The MSVC linker just packed the strings near our unrelated vtable.

--- What IS the crash class, then? ---

Given:
- Crash happens at post-menu-load timing (after 25 heroes + wallet
  data have arrived over WebSocket from ags)
- Timing varies by renderer (24s DX11 / 32s nullrhi / 64s D3D12) —
  something related to how frequently a subsystem ticks
- Class object size 15,504 bytes; parent 232 bytes; multi-inheritance
- Class holds sub-object arrays keyed by index (0, 1, 2 → 2 or 3
  entries each), a 768-byte hash table, 6 inline 0x38-byte slots
- Crashing method is a "get subarray by index, append value" pattern
- Class has smart-pointer (ref-counted) members (LOCK; xadd pattern)
- Only 2 live instances observed via findptr

New leading hypothesis: **The class is a UE 5.4 Slate / UMG / UI
subsystem** — something like FSlateApplication, FSlateWindowElementList,
or a per-viewport UI batch recorder. UI subsystems in UE 5.4 are:
- Large (multi-KB)
- Reference-counted
- Batched per-frame (tick-timing sensitive)
- Present in ALL renderer modes (including nullrhi, since the game engine
  still processes UI state even without rendering — the batch just isn't
  submitted to a GPU)

The 24s (DX11) / 32s (nullrhi) / 64s (D3D12) timing hierarchy is
consistent with UI batch buffers filling up at different rates depending
on frame rate: DX11 batches fastest (uncapped or higher-fps DX11
renderer), nullrhi mid, D3D12 slowest (capped by vsync or D3D12's
render-queue).

Alternative hypothesis: **The class is a UE 5.4 replication or
delegate-tracking subsystem** — something in the Actor Iris replication
graph or a delegate-notification list that ticks per frame. Also
consistent with the timing hierarchy.
