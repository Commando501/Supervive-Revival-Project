=== SESSION 40 — r.RayTracing=0 CVar OVERRIDE ALSO TRIGGERS ANTI-CHEAT ===

After `-noraytracing` was blocked by SUPERVIVE's anti-cheat cmdline
check, the recommendation was to use `-ini:Engine:[RendererSettings]:r.RayTracing=0`
which sets the same effect via CVar override (a path used extensively
already for AccelByte URL overrides in `launch-sv-open.bat`).

**Result: CVar-level raytracing disable ALSO triggers the anti-cheat
fail-fast. Same hardcoded `Rbp`/`R11`/`Rip` signature.**

on top of the working `launch-sv-open.bat` (which already uses `-ini:`

--- Implications ---

1. **Raytracing subsystem is anti-cheat-guarded.** SUPERVIVE requires
   raytracing to be enabled and detects its absence via multiple paths
   (cmdline flag, CVar setting, and probably in-game feature checks).
2. **`-ini:` CVar overrides ARE NOT safe from anti-cheat** — the
   anti-cheat's integrity check apparently examines actual runtime
   CVar state, not just cmdline parsing.
3. **We CANNOT disable raytracing to test whether it's the crash-holding
   subsystem** via any user-space method. Doing so triggers the fail-fast.
4. Timing of the fail-fast varies by trigger: 24s (DX11), 32s (nullrhi),
   42s (-noraytracing), 63s (r.RayTracing=0). This variability suggests
   different anti-cheat check points fire at different phases of init.
5. **The D3D12 mode natural crash (RVA +0x2976FF0, target class) may
   ALSO be an anti-cheat fail-fast** — we should re-examine that. In
   session 40 A1's dump, the crash was in target class with a real
   `this` pointer. But it's possible the anti-cheat check happens at
   the moment we're executing that class's method for other reasons,
   and BOTH cause the same crash signature. Need more careful analysis.
