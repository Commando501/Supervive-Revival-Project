INTERPRETATION OF N (this is the whole point of step 5)
-------------------------------------------------------
N is a NetIndex into SUPERVIVE's APlayerController ClassReps that our stock
stub ClassReps don't satisfy.
  - If N == 0: the client expects a replicated field at the very first index
    that our stock class doesn't have first. We must inject at least one
    replicated field ahead of whatever is currently ClassReps[0] on our side.
    (Session 36 saw N=0, but that was with the OLD path and no context; treat
    a fresh N=0 here as "add >=1 rep field at the front".)
  - If N > 0: N counts past the stock replicated fields our class already has.
    Cross-reference against the stock UE 5.4 APlayerController ClassReps order
    (SetUpRuntimeReplicationData preserves native declaration order) to find
    which SUPERVIVE-added field sits at that index. The strong Loki-addition
    candidates by naming (docs/session-22-schema-actor-loki-mods.txt) are
    ServerState, LokiReplicationStrategy, PoolableActorServerState.

Secondary outcome: if the client does NOT hit the FD3D12CommandContext crash
at +0x2976FF0 within ~90s and instead progresses, that alone is signal that
PC replication interacts with the D3D12 init path. Unlikely with no injection,
but note it if it happens.

If the client crashes the STUB, or the stub logs an array-index-out-of-bounds
on the PC replication path, that is the session-37 Option-A failure mode

EXACT MECHANISM OF THE ERROR (nailed down this session):
  DataReplication.cpp:1182 logs "ReceivedBunch: Invalid replicated field N"
  with N = FieldCache->FieldNetIndex. It fires in the post-RepLayout field
  loop, whose branches are:
    - FStructProperty  -> handled as a custom-delta property   (line 1095)
    - UFunction        -> handled as an RPC                     (line 1156)
    - anything else    -> "Invalid replicated field N"         (line 1180)
  IMPORTANT CONSEQUENCE: ordinary SCALAR replicated properties (bHidden,
  Role, TargetViewRotation, etc.) are consumed earlier by the RepLayout and
  do NOT appear in this loop. So N almost certainly resolves to either
  (a) a custom-delta STRUCT property, or (b) an RPC — OR the field-header
  section is being read at the wrong bit offset because our RepLayout property
  block is a different SIZE than the client expects (SUPERVIVE has extra
  CPF_Net props), which desyncs everything after it. The dump + real N
  disambiguates which.

STOCK UE 5.4 replicated-property inventory for the PC hierarchy (SET, from
headers; authoritative ORDER comes from the runtime dump — native classes use
field-iterator order, not offset sort, so don't assume header order):
  AActor (10): bReplicateMovement, bHidden, bTearOff, bCanBeDamaged,
               RemoteRole, AttachmentReplication, ReplicatedMovement, Owner,
               Role, Instigator
  AController (2): PlayerState, Pawn
  APlayerController (3): TargetViewRotation,
               AsyncPhysicsDataComponent_DEPRECATED (deprecated but CPF_Net),
               SpawnLocation
  => 15 replicated properties total across the chain, PLUS the per-level RPC
     UFunctions (many Server*/Client* on APlayerController), which take the
     higher indices at each level.

CLIENT-SIDE DIVERGENCE SOURCE (SUPERVIVE's AActor Loki additions, from
docs/session-22-schema-actor-loki-mods.txt; CPF_Net not dumped there so
replication is inferred by naming/type):
  - ServerState : StructProperty(PoolableActorServerState)  <-- PRIME SUSPECT.
      A struct named "ServerState" on AActor is almost certainly a replicated
      (likely custom-delta) struct. Stock AActor has no such field, so the
      client's NetCache carries an extra struct field our stub doesn't, which
      is exactly the kind of divergence that lands in the error loop above and
      shifts every later index. This is the first thing to inject in step 2.
  - LokiReplicationStrategy : StructProperty(LokiReplicationStrategy, 5 props:
      CustomReplicationStrategy enum, bOwningPlayer, bDistanceBased, bVision,
      bTeamBased) — a replication CONFIG struct; may or may not itself be
      CPF_Net. Second candidate.
  - Assorted bools/ints (bEnablePooling, bCanPrimeOnServer, MaxPoolSize, ...)
      — pooling config; less likely replicated, but the dump+N will tell.

So the leading hypothesis going into step 5: N will point at (or just past)
the position where SUPERVIVE's AActor.ServerState struct sits in the client's
NetCache, and step 2 = inject a matching replicated ServerState struct onto
AActor (NOT just APlayerController — this is an AActor-level addition, so it
shifts PC's indices via the FieldsBase chain). Confirm with the real N before
committing to that; do not inject speculatively (single-variable discipline).

Not implemented this session on purpose: without N we'd be guessing which
field(s) to inject, and the injection code is unverifiable until the live test

Add to unreal-stub/Source/Loki/Loki.cpp a parallel primitive to the existing
AppendToChildProperties() UFunction helpers, but targeting a UCLASS's
replicated-property list:

    static void AppendReplicatedPropertyToClass(UClass* Cls, FProperty* Prop)
    {
        // 1. Link Prop into Cls->ChildProperties (so it's a real FProperty on
        //    the class, addressable by RepLayout).
        // 2. Set CPF_Net on Prop->PropertyFlags.
        // 3. Append to Cls->ClassReps as an FRepRecord and assign
        //    Prop->RepIndex = Cls->ClassReps.Num()-1 (native classes keep
        //    declaration order; do NOT sort by offset — that's the blueprint
        //    path).
        // 4. Bump Cls->NetFields / ClassReps bookkeeping as
        //    SetUpRuntimeReplicationData() does.
    }
