INTERPRETATION OF N = 0
-----------------------
The error fires in the post-RepLayout field loop (DataReplication.cpp ~1050+),
whose valid branches are custom-delta STRUCT properties and RPC FUNCTIONS;
anything else -> "Invalid replicated field N". N = 0 is the FIRST field of the
class hierarchy. In stock, net index 0 = AActor.bReplicateMovement (a SCALAR),
which is NOT a valid target in this loop — so the client is reading the field
section at a MISALIGNED bit offset, i.e. our stub's RepLayout property block
for the PC is a DIFFERENT SIZE/SHAPE than the client's (SUPERVIVE-patched)
RepLayout expects. The desync manifests immediately (index 0), meaning the PC's
replicated schema diverges from the VERY FIRST replicated property.

This reproduces session 36's N=0 (noted in the pickup prompt) and confirms:
the empty-grid root cause is a PlayerController (and base-class) replicated-
schema divergence that begins at the front of the property list.

Consistent with session 22's AActor Loki additions: a replicated struct such as
ServerState (PoolableActorServerState) and/or LokiReplicationStrategy inserted
into AActor's CPF_Net set would change the RepLayout starting near the front and
desync everything after it.

CAVEAT (important for step 2): N=0 says "diverges at the front," not "append one
field." Native-class ClassReps use field-ITERATOR order, and our runtime
injection primitives APPEND to the tail (high RepIndex). To actually match
SUPERVIVE we must inject the missing replicated properties at the CORRECT
positions, not just add them at the end. Getting the order right is the hard
part flagged in docs/session-40-path-b-lite-analysis.txt (struct injection +
ordering). So step 2 is an iterative loop: inject candidate field(s), re-run
this exact probe, watch whether N moves / the connection survives past
ReceivedJoin, repeat.
