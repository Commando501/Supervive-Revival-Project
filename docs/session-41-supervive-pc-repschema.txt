WHY THIS EXPLAINS N=0
---------------------
The client's AActor RepLayout has 11 replicated cmds; our stub's has 10.
The RepLayout property block our stub sends is therefore a different
size/shape than the client's RepLayout expects, desyncing the read cursor.
By the time the client reaches the post-RepLayout field loop
(DataReplication.cpp ~1050), its cursor is misframed and the first field
header decodes to net index 0 -> a scalar (bReplicateMovement) that is not a
valid struct/function target -> "Invalid replicated field 0". One missing
replicated property at the AActor level is sufficient to cause this.

THE STRUCT: PoolableActorServerState  (2 members, simple)
---------------------------------------------------------
    State    : EnumProperty<UInt32Property>   (enum backed by uint32; 32 bits)
    Version  : UInt32Property                 (32 bits)
=> An 8-byte POD-ish struct. If it replicates member-wise (standard), the
   RepLayout expands ServerState into State (32b) + Version (32b) = 64 bits.
   OPEN QUESTION: whether PoolableActorServerState has a custom NetSerialize
   (WithNetSerializer), in which case RepLayout treats it as one opaque cmd.
   The CPF_RepNotify + no-CPF_RepSkip flags suggest standard member-wise
   replication, but confirm empirically in step 2 (inject, test, adjust if
   the wire size is off).

STEP 2 — now precisely scoped (was "8-15h", now much smaller)
-------------------------------------------------------------

TAIL-append on AActor (RepIndex 10, right after Instigator) on BOTH sides, so
appending it to our stub's AActor ClassReps lands at the identical index. No
mid-list insertion needed.
